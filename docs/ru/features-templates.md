# Шаблоны: определения, ссылки и YAML-якоря

Одна из самых распространённых проблем с инструментами Git-хуков — дублирование. Одна и та же команда линтинга попадает в `pre-commit`, та же команда сборки — в `pre-push`, и когда нужно изменить флаг линтера, приходится искать и обновлять его во всех хуках.

githops решает это на двух уровнях: **определения** для именованных переиспользуемых групп команд и **YAML-якоря** для совместного использования сырых фрагментов конфигурации.

## Определения

Определение — это именованная команда или список команд, объявленные под ключом верхнего уровня `definitions`. Любой хук может ссылаться на определение по имени вместо дублирования команды встроенно.

### Объявление определения

```yaml
definitions:
  lint:
    name: ESLint
    run: npx eslint . --ext .ts,.tsx

  typecheck:
    name: Проверка TypeScript
    run: npx tsc --noEmit
```

### Использование определения в хуке

```yaml
hooks:
  pre-commit:
    enabled: true
    commands:
      - $ref: lint
      - $ref: typecheck
        depends:
          - lint
```

При запуске хука githops раскрывает каждый `$ref` в команду, объявленную в `definitions`. Изменение значения `run` в определении обновляет все хуки, которые на него ссылаются.

### Передача дополнительных аргументов

Можно добавить дополнительные аргументы к команде `run` определения на месте вызова:

```yaml
hooks:
  pre-commit:
    commands:
      - $ref: lint
        args: "--fix"
```

Это запустит `npx eslint . --ext .ts,.tsx --fix` для данного хука, тогда как другие хуки, ссылающиеся на `lint` без `args`, запустят исходную команду.

### Переопределение отображаемого имени

Поле `name` в записи `$ref` переопределяет отображаемое имя в выводе githops и в графе:

```yaml
- $ref: lint
  name: "lint (с автоисправлением)"
  args: "--fix"
```

### Многокомандные определения

Определение может быть списком команд с собственными зависимостями:

```yaml
definitions:
  full-check:
    - name: install
      run: npm ci
    - name: lint
      run: npx eslint .
      depends:
        - install
    - name: test
      run: npm test
      depends:
        - lint
```

Ссылка `$ref: full-check` в хуке раскрывается в все три команды с сохранением графа зависимостей.

## YAML-якоря

Стандартные YAML-якоря работают в любом месте `githops.yaml`. Они разрешаются YAML-парсером до того, как githops читает файл, поэтому могут использоваться для любых структурных фрагментов — переменных окружения, наборов команд, вложенных опций.

### Совместное использование переменных окружения

```yaml
x-common-env: &common-env
  NODE_ENV: production
  CI: "true"

hooks:
  pre-push:
    commands:
      - name: build
        run: npm run build
        env: *common-env

  pre-commit:
    commands:
      - name: lint
        run: npm run lint
        env: *common-env
```

### Совместное использование блока команды

```yaml
x-lint: &lint-cmd
  name: lint
  run: npx eslint .

hooks:
  pre-commit:
    commands:
      - <<: *lint-cmd

  pre-merge-commit:
    commands:
      - <<: *lint-cmd
```

Ключ слияния `<<` — стандартный YAML и поддерживается всеми парсерами YAML 1.1, включая тот, что использует githops.

## Определения против YAML-якорей

Оба подхода решают проблему дублирования, но имеют разные компромиссы:

| | Определения | YAML-якоря |
|---|---|---|
| Видны в графе | Да | Нет |
| Ссылка по имени | Да | Нет (только структурные) |
| Поддержка `$ref` с `args` | Да | Нет |
| Работа с несколькими файлами | Нет (один файл) | Нет (один файл) |
| YAML-стандарт | Нет (расширение githops) | Да |
| Автодополнение в редакторе | Да (через схему) | Зависит от редактора |

Используйте определения, когда нужны именованные переиспользуемые команды, видимые в графе и допускающие ссылки с аргументами. Используйте YAML-якоря для сырого структурного совместного использования — повторяющихся словарей переменных окружения, общих наборов опций или фрагментов конфигурации, которые сами по себе не являются командами.
