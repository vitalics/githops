# Параллелизация

По умолчанию githops выполняет команды в хуке последовательно в порядке их объявления (с учётом ограничений `depends`). Для хуков, выполняющих несколько медленных проверок — проверку типов, линтинг, тесты — последовательное выполнение теряет время. Флаг `parallel` меняет это поведение.

## Включение параллельного выполнения

Установите `parallel: true` для любого хука:

```yaml
hooks:
  pre-push:
    enabled: true
    parallel: true
    commands:
      - name: lint
        run: npm run lint
      - name: typecheck
        run: npx tsc --noEmit
      - name: test
        run: npm test
```

С `parallel: true` githops запускает все команды одновременно и ждёт завершения всех, прежде чем позволить хуку завершиться. Если какая-либо команда завершается с ошибкой, хук завершается неудачно и Git-операция прерывается.

## Взаимодействие с `depends`

`parallel: true` не игнорирует `depends`. Команды, объявляющие зависимости, по-прежнему ждут их завершения перед запуском. Это значит, что в рамках одного хука можно иметь смесь параллельных и последовательных этапов:

```yaml
hooks:
  pre-push:
    enabled: true
    parallel: true
    commands:
      - name: install
        run: npm ci

      - name: lint
        run: npm run lint
        depends:
          - install

      - name: typecheck
        run: npx tsc --noEmit
        depends:
          - install

      - name: test
        run: npm test
        depends:
          - install
```

В этой конфигурации:

1. Сначала выполняется `install`.
2. После успешного завершения `install` параллельно запускаются `lint`, `typecheck` и `test`.
3. Хук успешно завершается только когда все три команды отработают без ошибок.

Это даёт двухэтапный конвейер: одиночная фаза установки, за которой следует полностью параллельная фаза проверок.

## Модель выполнения

githops строит граф зависимостей для всех команд хука. Команды без неразрешённых зависимостей готовы к запуску. При `parallel: true` все готовые команды запускаются одновременно. По мере завершения команд немедленно запускаются вновь ставшие доступными. Это продолжается до тех пор, пока все команды не выполнятся или какая-либо не завершится ошибкой.

При `parallel: false` (по умолчанию) готовые команды выполняются по одной в порядке объявления.

## Вывод

При параллельном выполнении githops добавляет к каждой строке вывода имя команды, чтобы можно было различить, какая команда что вывела:

```
[lint]      Running ESLint...
[typecheck] Running tsc...
[test]      Running Jest...
[lint]      Finished in 2.1s
[typecheck] error TS2345: Argument of type...
[test]      Finished in 4.8s
```

При ошибке команды githops выводит её полный вывод (включая stderr) и код завершения.

## Когда использовать параллелизацию

Параллельное выполнение наиболее ценно, когда:

- Хук запускает несколько независимых проверок (линтинг, проверка типов, тестирование).
- Каждая проверка занимает больше секунды-двух.
- Проверки не записывают данные в одни и те же файлы.

Избегайте `parallel: true`, когда команды должны записывать общее состояние в определённом порядке, или когда конкуренция за ресурсы (CPU, память, дисковый ввод-вывод) сделает параллельное выполнение медленнее последовательного.

## Анализ времени выполнения

Запустите githops с `--verbose`, чтобы увидеть время выполнения каждой команды:

```sh
githops check --verbose
```

Это помогает определить, какие команды являются узким местом, и стоит ли включать параллелизацию для данного хука.
