# Введение

## Что такое githops?

githops — это инструмент командной строки для управления Git-хуками через единый YAML-файл конфигурации. Вместо того чтобы хранить шелл-скрипты в `.git/hooks/` и надеяться, что каждый разработчик в команде настроил их вручную, вы описываете хуки в `githops.yaml`, коммитите этот файл и один раз запускаете `githops sync`. Все участники команды автоматически получают одинаковые хуки.

## Проблема

Git-хуки мощный инструмент. Они могут запускать линтеры перед коммитом, проверять сообщения коммитов, запускать тесты перед пушем и соблюдать соглашения проекта без ожидания результатов CI. Беда в том, что Git-хуки не коммитятся в репозиторий — директория `.git/hooks/` намеренно исключена из системы контроля версий.

На практике это приводит к следующим проблемам:

- **Несогласованность.** Новые члены команды вообще не получают хуки, потому что никто не сказал им настроить скрипты вручную.
- **Дублирование.** Одна и та же команда `npm run lint` оказывается скопирована в `pre-commit`, `pre-push` и иногда в `commit-msg`.
- **Хрупкость.** Шелл-скрипты в `.git/hooks/` не имеют структуры, порядка зависимостей и могут молча падать на путях Windows.
- **Непрозрачность.** Нет единого места, где можно увидеть, какие хуки активны, что они запускают и зачем.

Такие инструменты, как Husky, lefthook и pre-commit, решают часть этих проблем, но каждый из них идёт на компромиссы, которые могут не подойти вашему проекту. githops создан с другим набором приоритетов.

## Основные идеи

**Один файл — все хуки.** Каждый хук для каждого Git-события настраивается в `githops.yaml`. Достаточно прочитать один файл, чтобы понять всё, что выполняется в вашем репозитории.

**Определения для повторного использования.** Общие команды — запуск линтера, сборка проекта, проверка lockfile — можно вынести как именованные определения и ссылаться на них из нескольких хуков через `$ref`. Никакого копирования.

**YAML-якоря и шаблоны.** Стандартные YAML-якоря (`&anchor` / `*alias`) работают из коробки для совместного использования фрагментов конфигурации между хуками.

**Параллельное выполнение.** Флаг `parallel: true` на уровне хука запускает команды параллельно и ждёт завершения всех. Это важно для медленных линтеров или проверки типов.

**Умное кэширование.** Команды могут объявлять файлы, от которых зависят. githops пропускает команду, если ни один из этих файлов не изменился с момента последнего успешного запуска.

**Визуальный граф.** `githops graph` открывает браузерный интерфейс с графом зависимостей, где можно наглядно видеть, как хуки, команды и определения связаны между собой, и редактировать их интерактивно.

**Кросс-платформенность.** githops — единый бинарный файл на Rust без зависимостей от среды выполнения. Он работает на macOS, Linux и Windows без Node.js, Python или любого другого интерпретатора.

## Что выходит за рамки

githops не заменяет ваш CI-конвейер. Хуки, которыми он управляет, выполняются локально на машинах разработчиков. Для серверной проверки используйте вашу CI-систему — githops помогает синхронизировать локальные и CI-проверки за счёт общих определений команд, но сам в CI не запускается.
