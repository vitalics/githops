# githops против других инструментов

В этой области существует несколько инструментов. В таблице ниже они сравниваются по параметрам, которые обычно важны при выборе инструмента для реального проекта.

## Таблица сравнения

| Функция | githops | Husky | lefthook | pre-commit |
|---|---|---|---|---|
| Формат конфигурации | YAML | JSON / шелл-скрипты | YAML | YAML |
| Зависимость от среды выполнения | Нет (единый бинарный файл) | Node.js | Бинарный файл Go | Python |
| Кросс-платформенность | Да | Да | Да | Да |
| Параллельное выполнение | Флаг на уровне хука | Нет | Да | Нет |
| Порядок зависимостей | Да (`depends`) | Нет | Нет | Нет |
| Умное кэширование | Да (на основе хеша файлов) | Нет | Нет | Да (частично) |
| Переиспользуемые определения | Да (`$ref`) | Нет | Нет | Нет |
| YAML-якоря | Да | Нет | Нет | Нет |
| Визуальный граф | Да | Нет | Нет | Нет |
| Самообновление | Да (`githops self-update`) | Нет | Нет | Нет |
| Автодополнение в шелле | Да | Нет | Нет | Нет |
| JSON Schema для конфигурации | Да | Нет | Нет | Нет |
| Миграция с других инструментов | Да (`githops migrate`) | — | — | — |
| Установщик Windows MSI | Да | Нет | Нет | Нет |

## Примечания по каждому инструменту

### Husky

Husky — наиболее широко используемый менеджер Git-хуков в экосистеме JavaScript. Он работает, записывая небольшие шелл-скрипты в `.git/hooks/`, которые делегируют выполнение скриптам из репозитория. Главное преимущество — глубокая интеграция с `npm`/`pnpm` — хуки устанавливаются автоматически при `npm install` через скрипт жизненного цикла `prepare`.

Основное ограничение — Husky обеспечивает только инфраструктуру. Каждый файл хука — это обычный шелл-скрипт, который вы пишете сами, поэтому параллельного выполнения, порядка зависимостей, кэширования и повторного использования между хуками не предусмотрено. Кроме того, требуется Node.js, что создаёт проблему для не-JavaScript проектов.

### lefthook

lefthook — менеджер хуков на Go с YAML-файлом конфигурации. Поддерживает параллельное выполнение внутри хука и заметно быстрее Husky для больших репозиториев. После установки не требует зависимостей от среды выполнения.

По сравнению с githops, lefthook не имеет порядка зависимостей между командами внутри хука, контентного кэширования, переиспользуемых определений команд и визуальных инструментов. Хороший выбор для команд, которым нужно что-то лёгкое без полного набора функций.

### pre-commit

pre-commit ориентирован на запуск проверок из внешних репозиториев — вы объявляете «плагины» хуков, размещённые на GitHub, а pre-commit скачивает и запускает их в изолированных виртуальных окружениях. Это мощно для языково-независимых линтеров, которые не хочется устанавливать глобально.

Компромисс в том, что pre-commit требует Python и управляет собственным окружением для каждого плагина. Накладные расходы на запуск выше, а модель загрузки плагинов из удалённых репозиториев вводит сетевую зависимость. К тому же по умолчанию управляется только хук `pre-commit` — поддержка других хуков существует, но является второстепенной.

### githops

githops предлагает иной подход: это не система плагинов и не фреймворк. Это структурированный YAML-исполнитель для уже имеющихся у вас хуков. Вы пишете команды сами, организуете их с помощью определений и ссылок, а githops берёт на себя порядок выполнения, параллелизм и кэширование. Визуальный граф делает всю картину наглядной.

Если в вашем проекте уже есть команды, которые нужно запускать — линтинг, форматирование, проверка типов, тестирование — githops даёт структуру для управления ими во всех Git-хуках без добавления зависимости от среды выполнения.
