# Кэширование

githops поддерживает контентное кэширование для отдельных команд. Когда у команды есть конфигурация кэша, githops вычисляет хеш файлов, от которых зависит команда, и пропускает её, если хеш не изменился с момента последнего успешного запуска.

Это особенно полезно для медленных проверок — проверка типов, запуск тестов, шаги сборки — когда хочется избежать их повторного запуска при изменении несвязанных файлов.

## Включение глобального кэширования

Кэширование должно быть включено глобально, прежде чем конфигурации кэша отдельных команд возымеют эффект:

```yaml
cache:
  enabled: true
```

Опционально, измените директорию кэша (по умолчанию: `.githops/cache`):

```yaml
cache:
  enabled: true
  dir: .githops/cache
```

Добавьте `.githops/cache` в `.gitignore`:

```
.githops/cache
```

## Конфигурация кэша для команды

Добавьте блок `cache` к любой встроенной команде:

```yaml
hooks:
  pre-push:
    commands:
      - name: typecheck
        run: npx tsc --noEmit
        cache:
          inputs:
            - "src/**/*.ts"
            - "src/**/*.tsx"
            - tsconfig.json
          key:
            - package.json
```

### `inputs`

Список glob-шаблонов. githops хеширует содержимое всех совпадающих файлов. Если хеш совпадает с сохранённым хешем от последнего успешного запуска, команда пропускается.

```yaml
cache:
  inputs:
    - "src/**/*.ts"
    - "src/**/*.tsx"
```

### `key`

Необязательный список дополнительных файлов, содержимое которых включается в ключ кэша. Используйте для конфигурационных файлов, влияющих на вывод команды, но не охваченных `inputs`:

```yaml
cache:
  inputs:
    - "src/**/*.ts"
  key:
    - tsconfig.json
    - package-lock.json
```

Полный ключ кэша — SHA-256 хеш объединённого содержимого всех файлов из `inputs` и `key`.

## Как это работает

1. Перед запуском команды githops раскрывает все шаблоны из `inputs` и `key`.
2. Содержимое совпавших файлов хешируется алгоритмом SHA-256.
3. githops проверяет наличие маркерного файла в директории кэша с именем `<имя-команды>.<хеш>.ok`.
4. Если маркерный файл существует, команда пропускается и githops сообщает «cached».
5. Если маркерного файла нет, команда выполняется в обычном режиме.
6. При успешном завершении (код выхода 0) githops записывает маркерный файл.
7. При неудаче команды маркер не записывается, и команда будет выполнена снова в следующий раз.

## Инвалидация кэша

Кэш автоматически инвалидируется при изменении любого файла из `inputs` или `key`. Вручную очищать кэш для этого не нужно.

Для ручной очистки всех записей кэша:

```sh
githops cache clear
```

Или через графический интерфейс: откройте вкладку Cache и нажмите **Clear all**.

## Просмотр кэшированных записей

```sh
githops cache list
```

Выводит каждую кэшированную команду, её ключ и как давно она была кэширована.

## Совместное использование кэширования и параллелизации

Кэширование и параллельное выполнение работают вместе. В параллельном хуке githops проверяет кэш для каждой команды независимо. Кэшированные команды пропускаются немедленно, поэтому уже прошедшие проверки никогда не блокируют параллельное выполнение команд, которым действительно нужно работать.

## Пример: эффективный хук pre-push

```yaml
cache:
  enabled: true

hooks:
  pre-push:
    enabled: true
    parallel: true
    commands:
      - name: typecheck
        run: npx tsc --noEmit
        cache:
          inputs:
            - "src/**/*.ts"
            - "src/**/*.tsx"
          key:
            - tsconfig.json

      - name: test
        run: npx jest --passWithNoTests
        cache:
          inputs:
            - "src/**/*.test.ts"
            - "src/**/*.ts"
          key:
            - jest.config.ts
            - package.json

      - name: lint
        run: npx eslint src
        cache:
          inputs:
            - "src/**/*.ts"
            - "src/**/*.tsx"
          key:
            - .eslintrc.json
            - package.json
```

С такой конфигурацией пуш коммита, изменяющего только README, пропускает все три проверки, поскольку ни один из файлов `inputs` не изменился. Пуш с изменением TypeScript-файла запустит typecheck и test (зависящие от `.ts`-файлов), но может пропустить lint, если `.eslintrc.json` и изменённые `.ts`-файлы совпадают с кэшированным хешем.
